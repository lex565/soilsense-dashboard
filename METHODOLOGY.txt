================================================================================
  PHASE 1: ARCHITECTURE DESIGN
================================================================================

APPROACH
--------
Merged two research threads:
1. Maryline's thesis: 10-m resolution downscaling via Random Forest + Kriging
2. Merlyn's vision: Early warning system (SMS + mobile + dashboard)

Designed 4-layer operational architecture:
- Layer 1 (Data Input): SMAP + Sentinel-2 + CHIRPS + u-blox
- Layer 2 (Processing): RF downscaling + alert trigger logic
- Layer 3 (Validation): u-blox comparison + Table 19 metrics
- Layer 4 (Output): SMS + web dashboard + mobile app

WHY 4 LAYERS?
- Separates concerns (data ≠ processing ≠ validation ≠ delivery)
- Allows modular testing (each layer independent)
- Makes it clear to stakeholders (farmers see Layer 4 only)
- Hides complexity (Layers 1-3 backend, user sees only results)

RATIONALE
---------
- 10-m resolution: Aligns with smallholder plot size (0.5-2 ha)
- Independent validation: Proves field-scale accuracy (builds trust)
- Bilingual SMS: Reaches low-literacy farmers
- Offline-first: Works in areas without reliable internet

LAYER IMPLEMENTATIONS
---------------------

Layer 1 (Data Input):
    Mock data generation (SMAP, Sentinel-2, CHIRPS, u-blox)
    In production: Replace with actual API calls

Layer 2 (Processing):
    ml_engine.py:
    - load_mock_model(): Returns RF model metadata
    - predict_soil_moisture(smap, chirps_7d, ndvi, model):
      * Normalize inputs
      * Weighted combination (SMAP 60%, CHIRPS 15%, NDVI 15%, stress 10%)
      * Add kriging smoothness (std=0.0089)
      * Return 10-m predictions

    Alert trigger logic:
    - SM < 0.10 → drought_critical (🔴)
    - SM < 0.15 → irrigation_needed (🟡)
    - SM ≥ 0.15 → safe (🟢)

Layer 3 (Validation):
    validation.py:
    - validate_predictions(predictions, ublox_truth):
      * Calculate Pearson R (target: ≥0.80)
      * Calculate RMSE (target: ≤0.040)
      * Calculate MAE (target: ≤0.030)
      * Calculate Bias (target: ±0.02)
      * Return metrics dict

    - calculate_trust_score(metrics):
      * 🟢 all pass → high confidence
      * 🟡 some fail → medium confidence
      * 🔴 many fail → low confidence

Layer 4 (Output):
    sms_dispatcher.py:
    - generate_sms_alert(cell_id, sm_value, alert_type, crop):
      * Create bilingual templates (English + Tshivenda)
      * Ensure ≤160 characters (SMS standard)
      * Return formatted message

    - dispatch_sms(phone, message, gateway):
      * Prepare Twilio/Afrimotech integration
      * Return dispatch record with status

ERROR HANDLING
--------------
Added try/catch blocks to all endpoints:
    @app.post("/pipeline/run")
    def run_daily_pipeline():
        try:
            [execute pipeline]
            return {status: "success", ...}
        except Exception as e:
            print(f"✗ Pipeline error: {e}")
            return {status: "error", error: str(e)}

Graceful degradation:
- If SYSTEM_STATE["current_predictions"] is None → return empty data
- Frontend interprets empty response → falls back to mock data
- User sees "(Mock Mode)" indicator in button

FRONTEND ERROR HANDLING
---------
fetch(API_BASE + endpoint)
    .then(res => res.json())
    .catch(error => {
        console.error("API error:", error);
        return MOCK_DATA[endpoint];  // Fallback
    });

Async/await with try/catch:
    async function updateDashboard(useMock = false) {
        try {
            if (useMock || !BACKEND_AVAILABLE) {
                use MOCK_DATA;
            } else {
                fetch from API;
            }
        } catch (error) {
            use MOCK_DATA as fallback;
        }
    }

================================================================================
  KEY TECHNICAL DECISIONS & RATIONALE
================================================================================

1. OFFLINE-FIRST DESIGN
   Decision: Embed mock data, work without backend
   Rationale:
   - Test design before backend ready
   - Works in areas without internet
   - Graceful degradation if API fails
   - Better UX (instant load, no API latency)

2. LEAFLET.JS OVER MAPBOX
   Decision: Leaflet for map library
   Rationale:
   - Lightweight (leaflet.js is 40KB vs Mapbox 150KB)
   - Free tiles (Esri, OpenStreetMap)
   - Excellent for satellite imagery
   - Works offline with cached tiles
   - No API key required

3. VANILLA JS (NO REACT/FRAMEWORK)
   Decision: Pure JavaScript, no build step
   Rationale:
   - Simpler for prototyping
   - Faster loading (no bundle)
   - Easier to understand/modify
   - Works in offline scenarios
   - Can add React later if needed

4. MOCK DATA EMBEDDED
   Decision: MOCK_DATA object in app.js
   Rationale:
   - No external file dependencies
   - Self-contained prototype
   - Instant testing without setup
   - Easy to update/modify

5. FASTAPI (NOT DJANGO/FLASK)
   Decision: FastAPI for backend
   Rationale:
   - Automatic OpenAPI/Swagger docs (/docs)
   - Faster startup than Django
   - Built-in async support
   - Type hints make code clearer
   - Lightweight (one Python file can be full API)

6. 4-LAYER ARCHITECTURE (NOT MVC)
   Decision: Data → Processing → Validation → Output
   Rationale:
   - Matches operational workflow (real system has 4 stages)
   - Each layer testable independently
   - Hides Layers 1-3 from user (Layer 4 only visible)
   - Aligns with Maryline's thesis structure
   - Clearer to explain to farmers

7. DARK THEME (NOT LIGHT)
   Decision: bg-gray-900, text-white
   Rationale:
   - Matches disaster management dashboard pattern
   - Professional appearance
   - Reduces eye strain (long monitoring sessions)
   - High contrast (accessibility)
   - Modern UI standard
